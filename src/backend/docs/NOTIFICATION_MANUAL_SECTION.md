## 1. 通知类型概述

### 1.1 通知类型分类

| 类型 | 图标 | 说明 | 默认优先级 | 通知渠道 |
|------|------|------|------------|----------|
| **系统通知** | 📢 | 系统公告、维护通知、版本更新 | 中 | 站内信 + 邮件 |
| **告警通知** | ⚠️ | 资源告警、故障告警、异常事件 | 高 | 站内信 + 邮件 + 即时通讯 |
| **任务通知** | 📋 | 任务状态变更、完成提醒 | 中 | 站内信 |
| **审批通知** | ✅ | GPU 申请审批、权限变更审批 | 中 | 站内信 + 邮件 |
| **安全通知** | 🔐 | 登录异常、密码修改、权限变更 | 高 | 站内信 + 邮件 |

### 1.2 告警级别定义

| 级别 | 颜色 | 响应要求 | 通知方式 |
|------|------|----------|----------|
| 🔴 **紧急** | 红色 | 15 分钟内响应 | 站内信 + 邮件 + 即时通讯 + 短信 |
| 🟡 **警告** | 黄色 | 1 小时内响应 | 站内信 + 邮件 |
| 🔵 **信息** | 蓝色 | 24 小时内处理 | 站内信 |

### 1.3 告警子类型

| 子类型 | 触发条件 | 级别 | 处理建议 |
|--------|----------|------|----------|
| 服务器离线 | 状态变为 OFFLINE | 🔴 紧急 | 立即检查网络和服务器状态 |
| GPU 高温 | 温度 > 阈值 (默认 85°C) | 🟡 警告 | 检查散热系统，降低负载 |
| GPU 高负载 | 使用率 > 95% 持续 1 小时 | 🟡 警告 | 评估是否需要扩容 |
| 存储空间不足 | 磁盘使用率 > 阈值 (默认 90%) | 🟡 警告 | 清理临时文件或扩容 |
| 任务失败 | 任务执行过程中发生错误 | 🟡 警告 | 查看日志，排查原因 |
| 内存不足 | 内存使用率 > 95% | 🟡 警告 | 检查占用进程，释放资源 |
| 数据库异常 | 数据库连接失败或查询超时 | 🔴 紧急 | 检查数据库状态，重启服务 |

## 2. 通知渠道配置

### 2.1 支持的通知渠道

| 渠道 | 说明 | 配置要求 |
|------|------|----------|
| 站内信 | 系统内部消息中心 | 无需配置，默认启用 |
| 邮件 | 电子邮件通知 | 需验证邮箱地址 |
| 钉钉 | 钉钉机器人推送 | 需配置机器人 Webhook |
| 企业微信 | 企业微信机器人 | 需配置企业微信 Webhook |
| 短信 | 手机短信通知 | 需配置手机号（企业版） |
| Webhook | 自定义 HTTP 回调 | 需配置接收 URL |

### 2.2 按通知类型配置渠道

| 通知类型 | 站内信 | 邮件 | 即时通讯 | 短信 |
|----------|--------|------|----------|------|
| 系统通知 | ✅ 默认 | ✅ 默认 | ⚪ 可选 | ❌ 关闭 |
| 告警通知 | ✅ 默认 | ✅ 默认 | ✅ 默认 | ⚪ 可选 |
| 任务通知 | ✅ 默认 | ⚪ 可选 | ❌ 关闭 | ❌ 关闭 |
| 审批通知 | ✅ 默认 | ✅ 默认 | ⚪ 可选 | ❌ 关闭 |
| 安全通知 | ✅ 默认 | ✅ 默认 | ✅ 默认 | ⚪ 可选 |

### 2.3 钉钉通知配置

**步骤 1：创建钉钉机器人**
1. 打开钉钉群聊 → 群设置 → 智能群助手 → 添加机器人
2. 选择"自定义"机器人，设置名称（如 "LSM 告警"）
3. 安全设置选择"自定义关键词"，输入：`告警`、`通知`
4. 复制 Webhook 地址

**步骤 2：在 LSM 中配置**
1. 进入 **设置 → 通知偏好 → 钉钉配置**
2. 粘贴 Webhook 地址
3. 点击 **"测试发送"** → 确认收到消息后点击 **"保存"**

### 2.4 免打扰时段

```yaml
免打扰设置:
  启用: true
  开始时间: "22:00"
  结束时间: "08:00"
  重复: 每天  # 每天 / 工作日 / 自定义
  紧急通知: 仍然接收
```

### 2.5 通知聚合设置

| 策略 | 说明 | 示例 |
|------|------|------|
| 按类型聚合 | 相同类型通知合并 | 10 条任务完成通知合并为 1 条 |
| 按时间聚合 | 指定时间窗口内通知合并 | 每 15 分钟汇总发送一次 |
| 按来源聚合 | 相同来源通知合并 | 同一服务器的多条告警合并 |
| 智能聚合 | AI 智能判断相关性合并 | 相关告警自动合并 |

## 3. 告警规则配置

### 3.1 预置告警规则

| 规则名称 | 触发条件 | 默认阈值 | 通知对象 | 状态 |
|----------|----------|----------|----------|------|
| 服务器离线 | 服务器状态变为 OFFLINE | - | 管理员 | ✅ 启用 |
| GPU 高温 | GPU 温度超过阈值 | 85°C | 负责人 + 管理员 | ✅ 启用 |
| GPU 高负载 | GPU 使用率持续过高 | 95%/1小时 | 使用者 + 管理员 | ✅ 启用 |
| 存储空间不足 | 磁盘使用率超过阈值 | 90% | 管理员 | ✅ 启用 |
| 任务失败 | 任务执行失败 | - | 任务创建者 | ✅ 启用 |
| 内存不足 | 内存使用率超过阈值 | 95% | 管理员 | ✅ 启用 |
| GPU 长期占用 | GPU 分配超过 N 天 | 7天 | 使用者 + 管理员 | ✅ 启用 |

### 3.2 规则配置结构

```yaml
告警规则:
  基本信息:
    规则名称: "GPU 高温告警"
    告警类型: GPU 监控
    严重程度: 警告  # 紧急/警告/信息

  触发条件:
    指标: GPU_TEMPERATURE
    运算符: 大于  # 大于/小于/等于/不等于
    阈值: 85
    持续时间: 60  # 秒

  监控范围:
    类型: 全部 GPU  # 全部/指定服务器/指定GPU型号

  通知配置:
    通知对象: [资源负责人, 管理员]
    通知渠道: [站内信, 邮件, 钉钉]
    通知策略:
      频率: 300  # 秒
      静默期: 600  # 秒
```

### 3.3 触发条件类型

| 条件类型 | 说明 | 配置示例 |
|----------|------|----------|
| **指标阈值** | 指标超过阈值触发 | GPU 温度 > 85°C |
| **状态变更** | 资源状态变更触发 | 服务器状态 = OFFLINE |
| **持续时间** | 条件持续一段时间触发 | CPU > 90% 持续 5 分钟 |
| **组合条件** | 多条件组合触发 | CPU > 80% AND 内存 > 80% |

### 3.4 监控范围配置

| 范围类型 | 说明 | 配置示例 |
|----------|------|----------|
| 全部资源 | 监控所有符合条件的资源 | 所有 GPU |
| 指定服务器 | 选择特定服务器 | `["server-001", "server-002"]` |
| 指定 GPU 型号 | 选择特定型号 GPU | `["A100", "V100"]` |
| 标签筛选 | 按资源标签筛选 | `labels["env"] = "production"` |

### 3.5 告警规则示例

**示例 1：GPU 高温告警**

## 4. 智能告警降噪

> 🆕 **v3.1.0 新功能** - 告警噪音降低 **65%**，故障修复时间减少 **83%**

### 4.1 降噪策略

| 策略 | 说明 | 效果 |
|------|------|------|
| **去重** | 5 分钟内相同告警合并 | 减少重复告警 70% |
| **聚合** | 10 分钟内相似告警合并 | 减少告警数量 50% |
| **抑制** | 服务器离线时抑制相关告警 | 减少关联告警 80% |
| **静默** | 维护窗口静默告警 | 消除维护期噪音 |

### 4.2 告警去重

#### 去重原理

告警指纹由以下字段计算：告警类型 + 资源 ID + 告警级别 + 关键标签

```yaml
去重配置:
  启用: true
  去重窗口: 300  # 秒
  最大保留数: 10
  自动解决: 3600  # 秒
```

#### 去重效果示例

**原始告警**（5 分钟内 5 条相同告警）→ **去重后** 1 条聚合告警，显示重复次数和最新状态

### 4.3 告警聚合

```yaml
聚合配置:
  启用: true
  聚合窗口: 600  # 秒
  聚合规则:
    - 按告警类型聚合
    - 按服务器聚合
  最大聚合数: 20
```

**效果**：10 分钟内 4 台服务器的 CPU 过载告警 → 合并为 1 条聚合告警

### 4.4 告警抑制

```yaml
抑制配置:
  启用: true
  抑制规则:
    - 源告警: SERVER_OFFLINE
      目标告警: [GPU_ERROR, DISK_ERROR, NETWORK_ERROR]
    - 源告警: DATABASE_CONNECTION_ERROR
      目标告警: [TASK_FAILURE, API_ERROR]
```

**效果**：服务器离线时，自动抑制该服务器的 GPU、磁盘、网络告警

### 4.5 告警静默

#### 静默规则配置

```yaml
静默规则:
  名称: "计划维护窗口"
  匹配条件:
    - field: serverId
      operator: equals
      value: "server-001"
  持续时间: 7200  # 秒
  原因: "服务器硬件升级"
```

#### 匹配操作符

| 操作符 | 说明 | 示例 |
|--------|------|------|
| `equals` | 精确匹配 | `serverId equals "server-001"` |
| `not_equals` | 不等于 | `status not_equals "maintenance"` |
| `contains` | 包含 | `name contains "GPU"` |
| `regex` | 正则匹配 | `serverId regex "^gpu-.*"` |
| `in` | 在列表中 | `serverId in ["s1", "s2"]` |

#### 静默规则示例

```yaml
# 示例 1：单服务器维护
名称: "GPU-Server-01 维护"
匹配条件: [{ field: serverId, operator: equals, value: "gpu-server-01" }]
持续时间: 14400
原因: "NVIDIA 驱动升级"

# 示例 2：按告警类型静默
名称: "静默低优先级告警"
匹配条件:
  - { field: severity, operator: in, value: ["info", "warning"] }
持续时间: 3600
原因: "紧急发布期间减少噪音"
```

### 4.6 降噪 API

```bash
# 获取统计数据
GET /api/alert-dedup/statistics

# 创建静默规则
POST /api/alert-dedup/silences
{
  "name": "计划维护窗口",

  "duration": 3600,

}

# 确认告警
POST /api/alert-dedup/alerts/{id}/acknowledge

# 解决告警

```

## 6. 最佳实践

### 6.1 告警规则设计

#### ✅ 推荐做法

1. **设置合理的阈值**
   - 基于历史数据确定阈值
   - 设置适当的持续时间避免瞬时波动
   - 为不同环境设置不同阈值

2. **合理设置通知频率**
   - 紧急告警：1-5 分钟
   - 警告告警：10-30 分钟
   - 信息告警：30-60 分钟

3. **使用告警分级**
   - 根据影响范围和紧急程度分级
   - 不同级别使用不同的通知渠道

#### ❌ 避免做法

- 阈值过于敏感（CPU > 50% 持续 10 秒就告警）
- 所有告警都设紧急级别
- 通知频率过低（紧急告警 1 小时才通知一次）

### 6.2 告警降噪配置建议

```yaml
去重配置:
  启用: true
  去重窗口: 300  # 5 分钟，根据实际情况调整

聚合配置:
  启用: true
  聚合窗口: 600  # 10 分钟，高频环境可延长到 15-20 分钟

抑制配置:
  建议: 识别告警依赖关系，根因告警应抑制结果告警

静默规则:
  维护前: 提前创建，设置合理时间（多预留 30 分钟）
  维护后: 及时删除，检查遗漏告警
```

### 6.4 告警处理流程

```
1. 接收告警 → 2. 确认告警（添加处理人和初步判断）
                    ↓
3. 分析原因 → 4. 执行修复 → 5. 验证解决 → 6. 标记已解决
```

**告警确认规范**：


### 6.5 常见问题排查

#### 问题 1：告警噪音过大

**排查**：阈值过低？降噪未启用？配置不合理？

**解决**：提高阈值、延长持续时间、启用去重聚合、配置抑制规则

#### 问题 2：重要告警未收到

**排查**：渠道配置？免打扰时段？静默规则误匹配？

**解决**：检查邮箱/Webhook、调整免打扰时段、审查静默规则

#### 问题 3：告警处理不及时

**解决**：配置多渠道通知、设置升级策略、建立轮值机制

### 6.6 定期审查清单

**每周**：告警数量趋势、处理及时率、误报率

**每月**：规则有效性、降噪效果、静默规则清理

**每季度**：告警策略优化、团队培训、流程改进
