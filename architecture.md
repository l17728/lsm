# 实验室服务器管理系统 (LSM) - 系统架构设计

## 1. 技术选型

### 1.1 前端框架
**选型：React 18 + TypeScript**

**理由：**
- 生态成熟，组件库丰富（Ant Design、Material UI）
- 适合构建复杂的管理后台界面
- TypeScript 提供强类型支持，降低维护成本
- 支持 Server Components，优化首屏加载
- 社区活跃，人才储备充足

**技术栈：**
- React 18 (并发渲染)
- TypeScript 5.x
- Zustand/Redux Toolkit (状态管理)
- React Query (数据获取)
- Ant Design Pro (UI 组件)
- Vite (构建工具)

### 1.2 后端框架
**选型：Go 1.21 + Gin Framework**

**理由：**
- 高性能，适合高并发场景（2000+ 节点）
- 原生协程支持，并发模型简单高效
- 编译型语言，部署简单，资源占用低
- 适合系统级编程和基础设施开发
- 优秀的监控和可观测性生态

**技术栈：**
- Go 1.21+
- Gin (Web 框架)
- gRPC (服务间通信)
- Wire (依赖注入)
- Zap (日志)
- Prometheus Client (指标采集)

### 1.3 数据库
**选型：PostgreSQL 15 + Redis 7**

**PostgreSQL (主数据库)：**
- 支持复杂查询和事务
- JSONB 类型支持灵活数据结构
- 优秀的并发控制 (MVCC)
- 成熟的分区表支持 (应对海量数据)
- 强大的扩展生态 (TimescaleDB for 时序数据)

**Redis (缓存/会话)：**
- 高性能缓存 (热点数据)
- 会话存储
- 分布式锁
- 发布/订阅 (实时通知)
- 有序集合 (排行榜、队列)

### 1.4 消息队列
**选型：Apache Kafka**

**理由：**
- 高吞吐量 (百万级消息/秒)
- 持久化存储，支持消息回溯
- 优秀的水平扩展能力
- 适合日志采集和事件驱动架构
- 成熟的生态 (Kafka Connect, KSQL)

**使用场景：**
- 服务器状态上报
- 任务执行日志
- 监控指标采集
- 异步任务处理
- 审计日志

### 1.5 容器编排
**选型：Kubernetes 1.28+**

**理由：**
- 行业标准，生态完善
- 强大的自动扩缩容 (HPA/VPA)
- 服务发现和负载均衡
- 配置管理和密钥管理
- 成熟的监控方案 (Prometheus + Grafana)

**部署架构：**
- 控制平面：3 节点高可用
- 工作节点：按需扩展
- Ingress: Nginx Ingress Controller
- Service Mesh: Istio (可选，用于复杂流量管理)

### 3.2 高并发资源请求处理

#### 并发控制策略

```go
// 资源分配并发控制伪代码
type ResourceManager struct {
    // 分布式锁 (Redis)
    locker *redislock.Client
    // 资源池
    pool *ResourcePool
    // 限流器
    limiter *rate.Limiter
}

func (rm *ResourceManager) AllocateServer(userID string, spec ResourceSpec) (*Server, error) {
    // 1. 用户级限流 (防止单个用户滥用)
    if !rm.limiter.Allow() {
        return nil, ErrRateLimited
    }
    
    // 2. 获取分布式锁 (防止超卖)
    lock, err := rm.locker.Lock("resource:allocate", 5*time.Second)
    if err != nil {
        return nil, ErrLockFailed
    }
    defer lock.Unlock()
    
    // 3. 检查资源池
    server := rm.pool.FindAvailable(spec)
    if server == nil {
        return nil, ErrNoResource
    }
    
    // 4. 标记为已分配 (数据库事务)
    err = rm.markAsAllocated(server.ID, userID)
    if err != nil {
        return nil, err
    }
    
    return server, nil
}
```

#### 性能优化

| 优化点 | 方案 | 预期效果 |
|--------|------|---------|
| 缓存热点数据 | Redis 缓存服务器状态 | 减少 90% DB 查询 |
| 批量操作 | 批量分配/释放接口 | 减少 80% 网络开销 |
| 异步处理 | Kafka 解耦非关键路径 | 提升 3 倍吞吐量 |
| 连接池 | DB/Redis 连接复用 | 减少连接建立开销 |
| 索引优化 | 覆盖索引 + 分区表 | 查询速度提升 10 倍 |

### 6.2 故障恢复策略

| 故障类型 | 检测方式 | 恢复策略 | RTO | RPO |
|---------|---------|---------|-----|-----|
