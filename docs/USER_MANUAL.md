### 第 1 章 系统概述

#### 1.1 什么是 LSM？

**LSM (Laboratory Server Management System)** 是一个全面的实验室服务器资源管理平台，专为科研实验室设计，用于高效管理服务器资源、GPU 分配和任务调度。

**核心价值**:
- 🎯 **资源优化** - 最大化 GPU/CPU 利用率，减少闲置浪费
- 📊 **透明管理** - 实时查看所有资源状态和使用情况
- ⚡ **高效调度** - 基于优先级的智能任务调度系统
- 🔒 **安全可靠** - 企业级安全认证和审计日志
- 📱 **随时随地** - 响应式设计，支持移动端访问

#### 1.2 适用场景

| 场景 | 说明 |
|------|------|
| 🏫 高校实验室 | 管理多台 GPU 服务器，分配给学生使用 |
| 🔬 研究机构 | 调度计算资源，支持多个研究项目 |
| 🏢 企业研发部门 | 管理 AI/ML 训练任务，优化资源使用 |
| 🧪 数据中心 | 监控服务器健康状态，及时处理故障 |

#### 1.3 核心功能

| 功能模块 | 说明 | 主要用户 |
|---------|------|----------|
| 🖥️ **服务器管理** | 管理多台服务器，实时跟踪状态 | ADMIN, MANAGER |

#### 1.5.3 手动部署

**适合需要自定义配置的场景**。

**Step 1: 克隆项目**

```bash
git clone https://github.com/l17728/lsm.git
cd lsm
```

**Step 2: 配置环境变量**

```bash
# 复制配置模板
cp .env.example .env

# 编辑配置文件
vim .env
```

**关键配置项**:

```env
# 数据库密码（必须修改）
DB_PASSWORD=your_secure_password

# Redis 密码（必须修改）
REDIS_PASSWORD=your_redis_password

# JWT 密钥（必须修改，生成命令见下）
JWT_SECRET=your_jwt_secret_key

# 前端 API 地址
VITE_API_BASE_URL=http://your-server-ip:8080/api
```

**生成安全密钥**:

```bash
# JWT 密钥
openssl rand -base64 64

# 数据库密码
openssl rand -base64 32
```

**Step 3: 启动服务**

```bash
# 启动所有服务
docker-compose up -d

# 查看服务状态
docker-compose ps

# 查看日志
docker-compose logs -f
```

**Step 4: 数据库初始化**

```bash
# 运行数据库迁移
docker-compose exec backend npx prisma migrate deploy

# 生成 Prisma 客户端
docker-compose exec backend npx prisma generate
```

**Step 5: 验证部署**

```bash
# 检查后端健康状态
curl http://localhost:8080/api/health
# 预期输出：{"success":true,"data":{"status":"ok"}}

# 检查前端
curl http://localhost:3000
```

#### 1.5.4 生产环境部署

**使用生产配置文件**:

```bash
# 使用生产 Docker Compose 配置
docker-compose -f docker-compose.prod.yml up -d

# 或复制配置
cp docker-compose.prod.yml docker-compose.override.yml
docker-compose up -d
```

**生产环境检查清单**:

- [ ] 修改所有默认密码（DB、Redis、JWT、Grafana）
- [ ] 配置 HTTPS/SSL 证书
- [ ] 设置防火墙规则（仅开放 80、443 端口）
- [ ] 配置邮件服务 SMTP
- [ ] 设置备份计划
- [ ] 启用监控告警

**防火墙配置** (Ubuntu):

```bash
# 允许 SSH
sudo ufw allow 22/tcp

# 允许 HTTP/HTTPS
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# 启用防火墙
sudo ufw enable
```

#### 1.5.5 常见问题

**Q: Docker 未安装怎么办？**

```bash
# Ubuntu/Debian
curl -fsSL https://get.docker.com | sh
sudo systemctl enable docker
sudo systemctl start docker

# 安装 Docker Compose
sudo curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose
sudo chmod +x /usr/local/bin/docker-compose
```

**Q: 端口被占用怎么办？**

```bash
# 查看端口占用
sudo netstat -tlnp | grep :8080

# 停止占用进程
sudo kill -9 <PID>

# 或修改 .env 中的端口配置
BACKEND_PORT=8081
FRONTEND_PORT=3001
```

**Q: 数据库连接失败？**

```bash
# 检查 PostgreSQL 是否运行
docker-compose ps postgres

# 查看数据库日志
docker-compose logs postgres

# 重启数据库
docker-compose restart postgres
```

**Q: 如何更新到最新版本？**

```bash
# 拉取最新代码
git pull origin main

# 重新构建并启动
docker-compose down
docker-compose build
docker-compose up -d

# 运行数据库迁移
docker-compose exec backend npx prisma migrate deploy
```

#### 6.2 任务状态说明

##### 6.2.2 状态转换流程

```
                    │   创建任务   │
```

##### 6.2.3 状态详细说明

**待处理 (PENDING)**

- **含义**: 任务已成功创建，正在等待系统分配资源
- **持续时间**: 取决于资源可用性和任务优先级
- **可执行操作**: 编辑、取消、调整优先级
- **常见原因**:
  - 无可用 GPU 资源
  - 优先级较低，排队等待
  - 指定的服务器处于维护状态

**排队中 (QUEUED)**

- **含义**: 资源已分配，任务在执行队列中等待
- **持续时间**: 通常较短（秒级到分钟级）
- **可执行操作**: 取消
- **说明**: 此状态表示任务即将开始执行

**运行中 (RUNNING)**

- **含义**: 任务正在 GPU 服务器上执行
- **持续时间**: 取决于任务复杂度和数据量
- **可执行操作**: 取消、查看日志
- **监控内容**: 进度、资源使用、日志输出

**已完成 (COMPLETED)**

- **含义**: 任务执行成功，结果已保存
- **持续时间**: 永久（直到手动删除）
- **可执行操作**: 查看结果、下载文件、删除、重新运行
- **说明**: 可查看完整执行日志和输出文件

**已失败 (FAILED)**

- **含义**: 任务执行过程中发生错误
- **持续时间**: 永久（直到手动删除）
- **可执行操作**: 查看错误日志、重新运行、删除
- **常见失败原因**:

| 错误类型 | 说明 | 解决方案 |
|----------|------|----------|
| 资源不足 | GPU 显存或内存不足 | 减少批量大小或申请更多资源 |
| 脚本错误 | 代码存在 bug | 检查日志，修复代码 |
| 数据错误 | 数据集路径错误或格式问题 | 检查数据路径和格式 |
| 超时 | 执行时间超过限制 | 优化代码或申请更长执行时间 |
| 权限错误 | 文件或目录权限不足 | 检查文件权限设置 |

**已取消 (CANCELLED)**

- **含义**: 用户主动终止任务
- **持续时间**: 永久（直到手动删除）
- **可执行操作**: 删除、重新运行
- **说明**: 运行中取消会触发资源清理，可能需要等待

##### 6.3.3 优先级设置

优先级决定了任务获取资源的顺序，取值范围为 0-10，数值越大优先级越高。

| 优先级 | 级别 | 适用场景 | 示例 |
|--------|------|----------|------|
| 0-2 | 低 | 后台任务、非紧急任务 | 数据预处理、模型推理 |
| 3-4 | 较低 | 常规实验任务 | 模型调参、小规模训练 |
| 5-6 | 中等 | 标准任务（默认） | 常规训练任务 |
| 7-8 | 较高 | 重要任务 | 关键实验、论文实验 |
| 9-10 | 最高 | 紧急任务 | 截止日期任务、紧急修复 |

**优先级调度规则**:

```
1. 同优先级任务按创建时间先进先出 (FIFO)
2. 高优先级任务优先获取资源
3. 运行中的任务不会被抢占
4. 管理员可手动调整任务优先级
```

**设置建议**:

- ✅ 根据任务紧急程度合理设置
- ✅ 避免所有任务都设为最高优先级
- ✅ 长时间任务可设置较低优先级
- ❌ 不要滥用高优先级

```bash
export CUDA_VISIBLE_DEVICES=0,1,2,3
export PYTHONPATH=/workspace

# 创建输出目录
mkdir -p /output/$(date +%Y%m%d)

# 执行训练
python train.py \
    --model resnet50 \
    --data /data/imagenet \
    --output /output/$(date +%Y%m%d) \
    --epochs 100 \
    --batch-size 256

# 检查结果
if [ $? -eq 0 ]; then
    echo "训练完成！"
    python evaluate.py --model /output/$(date +%Y%m%d)/best_model.pth
else
    echo "训练失败！"
    exit 1
fi
```

##### 6.9.2 任务队列机制

**队列结构**:

| 队列类型 | 说明 | 优先级范围 |
|----------|------|------------|
| **高优先级队列** | 紧急任务、生产任务 | CRITICAL (3) |
| **标准队列** | 常规任务、日常实验 | HIGH (2), MEDIUM (1) |
| **低优先级队列** | 后台任务、批处理任务 | LOW (0) |

**队列调度流程**:

```
任务提交 → 进入对应优先级队列 → 等待资源 → 调度器评估 → 分配资源 → 执行
              │                     │
              ▼                     ▼
         ┌─────────┐          ┌─────────────┐
         │ 高优先级 │          │ 检查资源    │
         │ 队列    │          │ 可用性      │
         ├─────────┤          ├─────────────┤
         │ 标准队列 │    →     │ 评估服务器  │
         ├─────────┤          │ 负载情况    │
         │ 低优先级 │          │ 匹配 GPU    │
         │ 队列    │          │ 需求        │
         └─────────┘          └─────────────┘
```

**队列管理操作**:

| 操作 | 说明 |
|------|------|
| 查看队列深度 | 显示各优先级队列的等待任务数 |
| 查看预计等待时间 | 基于当前负载估算任务开始时间 |
| 查看资源占用 | 显示当前正在执行的任务及其资源使用 |

##### 6.9.3 智能调度策略

LSM 系统采用 AI 驱动的智能调度，综合考虑多个因素做出最优调度决策。

**调度决策因素**:

| 因素 | 权重 | 说明 |
|------|------|------|
| **任务优先级** | 25% | 高优先级任务优先调度 |
| **资源匹配度** | 20% | 任务需求与服务器资源的匹配程度 |
| **负载均衡** | 20% | 避免单点过载，均衡分配负载 |
| **用户公平性** | 15% | 确保各用户公平获取资源 |
| **能效优化** | 10% | 优先选择能耗效率高的服务器 |
| **可靠性** | 10% | 优先选择高可靠性服务器 |

**调度算法详解**:

**1. 优先级调度 (Priority Scheduling)**

- 高优先级任务始终优先于低优先级任务
- 相同优先级按提交时间先进先出 (FIFO)
- 支持优先级抢占（需管理员配置）

```
优先级排序示例:
队列中任务: T1(P=9), T2(P=7), T3(P=6), T4(P=5), T5(P=3)

调度顺序: T1 → T2 → T3 → T4 → T5
```

**2. 资源匹配调度 (Resource Matching)**

| 任务需求 | 匹配策略 |
|----------|----------|
| GPU 显存 ≤ 16GB | 优先分配 RTX 3090、V100 |
| GPU 显存 16-40GB | 分配 A100-40GB、V100-32GB |
| GPU 显存 > 40GB | 分配 A100-80GB |
| 多卡需求 | 分配同一服务器的多块 GPU |

**3. 负载均衡调度 (Load Balancing)**

```
负载均衡示例:
服务器 A: CPU 85%, GPU 90%  → 负载过高，避免分配新任务
服务器 B: CPU 30%, GPU 20%  → 负载较低，优先分配
服务器 C: CPU 50%, GPU 60%  → 负载适中，可分配
```

**4. 公平调度 (Fair Scheduling)**

```
公平调度计算:
用户 A: 3 个任务运行中 → 降低新任务优先级
用户 B: 0 个任务运行中, 等待 2 小时 → 提升任务优先级
用户 C: 1 个任务运行中 → 正常调度
```

**5. 预测性调度 (Predictive Scheduling)**

```
预测调度示例:
当前时间: 14:00
预测: 服务器 A 负载持续上升，预计 16:00 过载
决策: 将新任务调度到服务器 B，避免拥堵
```

##### 6.9.4 调度监控与调优

**监控指标**:

| 指标 | 说明 | 健康范围 |
|------|------|----------|
| 队列深度 | 等待任务数量 | < 50 |
| 平均等待时间 | 任务从提交到开始的时间 | < 30 分钟 |
| 调度成功率 | 成功分配资源的任务比例 | > 95% |
| 资源利用率 | 集群整体资源利用率 | 60-80% |
| 负载均衡度 | 各服务器负载差异 | < 20% |

#### 6.10 任务依赖关系

任务依赖允许定义任务之间的执行顺序，确保前置任务完成后才执行后续任务。

##### 6.10.1 依赖类型

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| **顺序依赖** | 任务 B 必须在任务 A 完成后执行 | 数据预处理 → 模型训练 |
| **并行依赖** | 任务 C 等待任务 A 和 B 都完成 | 多模型训练 → 结果合并 |
| **条件依赖** | 根据前置任务结果决定是否执行 | 训练成功 → 部署；训练失败 → 调参 |

##### 6.10.2 配置任务依赖

**方式一：创建任务时指定**

```yaml
# 任务配置示例
任务名称: 模型训练
依赖任务: 
  - taskId: "data-preprocess-task-id"
    type: "sequential"  # 顺序依赖
```

**方式二：在任务详情页添加依赖**

1. 进入任务详情页
2. 点击 **"添加依赖"** 按钮
3. 选择前置任务
4. 选择依赖类型
5. 点击 **"保存"**

##### 6.10.3 依赖关系可视化

```
依赖关系图示例:

┌──────────────┐
│ 数据预处理    │ (已完成 ✅)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 模型训练     │ (运行中 ▶️)
└──────┬───────┘
       │
       ├─────────────────┐
       ▼                 ▼
┌──────────────┐  ┌──────────────┐
│ 模型评估     │  │ 模型部署     │
│ (等待中 ⏳)  │  │ (等待中 ⏳)  │
└──────────────┘  └──────────────┘
```

##### 6.10.4 依赖执行策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **严格依赖** | 前置任务必须成功 | 关键任务链 |
| **宽松依赖** | 前置任务完成即可（无论成功失败） | 可容错的任务 |
| **条件依赖** | 根据前置任务输出决定执行 | 分支流程 |

### 第 8 章 通知中心

通知中心是 LSM 系统的消息枢纽，负责收集、分发和管理各类系统消息。通过通知中心，用户可以及时了解系统状态、任务进展和异常告警，确保高效运维和快速响应。

#### 8.1 通知类型说明

LSM 系统支持多种类型的通知，每种类型对应不同的业务场景和优先级。

##### 8.1.1 通知类型分类

| 类型 | 图标 | 说明 | 优先级 | 默认渠道 |
|------|------|------|--------|----------|
| **系统通知** | 📢 | 系统公告、维护通知、版本更新 | 中 | 站内信 + 邮件 |
| **告警通知** | ⚠️ | 资源告警、故障告警、异常事件 | 高 | 站内信 + 邮件 + 钉钉 |
| **任务通知** | 📋 | 任务状态变更、完成通知、失败提醒 | 中 | 站内信 |
| **审批通知** | ✅ | GPU 申请审批、权限变更审批 | 中 | 站内信 + 邮件 |
| **安全通知** | 🔐 | 登录异常、密码修改、权限变更 | 高 | 站内信 + 邮件 |

##### 8.1.2 系统通知

系统通知由管理员或系统自动触发，用于广播重要信息。

**系统通知子类型**:

| 子类型 | 触发场景 | 示例内容 |
|--------|----------|----------|
| 系统公告 | 管理员发布重要公告 | "系统将于今晚 22:00 进行升级维护" |
| 维护通知 | 计划维护前后 | "服务器 GPU-Server-01 即将进入维护模式" |
| 版本更新 | 系统版本更新时 | "LSM v3.2.0 已发布，新增功能..." |
| 服务状态 | 服务启停、重启 | "监控服务已重启，预计耗时 5 分钟" |

**系统通知特性**:
- 可设置置顶显示
- 支持富文本格式
- 可指定接收范围（全体用户/特定角色/特定用户）

##### 8.1.3 告警通知

告警通知是运维的核心，需要及时处理以保障系统稳定运行。

**告警通知子类型**:

| 子类型 | 触发条件 | 紧急程度 | 处理建议 |
|--------|----------|----------|----------|
| 服务器离线 | 服务器状态变为 OFFLINE | 🔴 紧急 | 立即检查网络和服务器状态 |
| GPU 高温 | GPU 温度 > 设定阈值 (默认 85°C) | 🟡 警告 | 检查散热系统，降低负载 |
| GPU 高负载 | GPU 使用率 > 95% 持续 1 小时 | 🟡 警告 | 评估是否需要扩容 |
| 存储空间不足 | 磁盘使用率 > 设定阈值 (默认 90%) | 🟡 警告 | 清理临时文件或扩容 |
| 任务失败 | 任务执行过程中发生错误 | 🟡 警告 | 查看日志，排查原因 |
| 内存不足 | 内存使用率 > 95% | 🟡 警告 | 检查占用进程，释放资源 |
| 数据库异常 | 数据库连接失败或查询超时 | 🔴 紧急 | 检查数据库状态，重启服务 |
| 网络异常 | 网络延迟 > 阈值或丢包 | 🟡 警告 | 检查网络设备和配置 |

**告警级别说明**:

| 级别 | 颜色 | 响应时间要求 | 通知方式 |
|------|------|-------------|----------|
| 🔴 紧急 | 红色 | 15 分钟内响应 | 站内信 + 邮件 + 钉钉 + 短信 |
| 🟡 警告 | 黄色 | 1 小时内响应 | 站内信 + 邮件 |
| 🔵 信息 | 蓝色 | 24 小时内处理 | 站内信 |

##### 8.4.2 预置告警规则

系统预置以下告警规则，可根据需求启用或禁用：

| 规则名称 | 触发条件 | 默认阈值 | 通知对象 | 默认状态 |
|----------|----------|----------|----------|----------|
| 服务器离线 | 服务器状态变为 OFFLINE | - | 管理员 | ✅ 启用 |
| GPU 高温 | GPU 温度超过阈值 | 85°C | 服务器负责人 + 管理员 | ✅ 启用 |

##### 8.4.3 创建自定义告警规则

**步骤**:

1. 点击 **"+ 新建规则"** 按钮
2. 填写基本信息：

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| 规则名称 | ✅ | 规则名称，简洁明了 | "A100 GPU 高温告警" |
| 规则描述 | ❌ | 详细描述规则用途 | "针对 A100 GPU 的高温告警" |
| 告警类型 | ✅ | 选择告警类型 | 服务器/GPU/任务/存储 |
| 严重程度 | ✅ | 紧急/警告/信息 | 警告 |

3. 配置触发条件：

| 条件类型 | 说明 | 配置示例 |
|----------|------|----------|
| 指标阈值 | 指标超过阈值触发 | GPU 温度 > 85°C |
| 状态变更 | 资源状态变更触发 | 服务器状态 = OFFLINE |
| 持续时间 | 条件持续一段时间触发 | CPU > 90% 持续 5 分钟 |
| 组合条件 | 多条件组合触发 | CPU > 80% AND 内存 > 80% |

4. 选择监控范围：

| 范围类型 | 说明 |
|----------|------|
| 全部资源 | 监控所有符合条件的资源 |
| 指定服务器 | 选择特定服务器 |
| 指定 GPU 型号 | 选择特定型号 GPU |
| 标签筛选 | 按资源标签筛选 |

5. 配置通知对象：

| 对象类型 | 说明 |
|----------|------|
| 管理员 | 发送给所有 ADMIN 角色用户 |
| 资源负责人 | 发送给资源的负责人 |
| 指定用户 | 手动选择特定用户 |
| 指定角色 | 选择 ADMIN/MANAGER/USER |
| 用户组 | 选择预定义的用户组 |

6. 配置通知渠道：

| 渠道 | 说明 | 适用场景 |
|------|------|----------|
| 站内信 | 系统内消息 | 所有通知 |
| 邮件 | 电子邮件 | 重要通知 |
| 钉钉 | 钉钉机器人 | 紧急告警 |
| 短信 | 手机短信 | 紧急告警（需配置） |

7. 配置通知策略：

| 策略 | 说明 |
|------|------|
| 通知频率 | 重复告警的通知间隔（默认 5 分钟） |
| 静默期 | 告警恢复后的静默时间（默认 10 分钟） |
| 升级策略 | 未处理时升级通知 |

8. 点击 **"保存并启用"**

#### 13.1 资源管理最佳实践

##### 13.1.1 GPU 分配策略

**按需申请原则**:

✅ **推荐做法**:
- 根据任务实际需求申请 GPU 数量和显存大小
- 小型实验先用单卡验证，确认可行后再申请多卡
- 任务完成后立即释放 GPU 资源
- 在任务描述中注明预计完成时间

❌ **避免做法**:
- 一次性申请所有可用 GPU "以防万一"
- 任务结束后忘记释放，导致资源长期占用
- 不说明使用目的，随意申请高端 GPU

**示例场景**:

| 任务类型 | 推荐 GPU 配置 | 申请理由示例 |
|---------|-------------|-------------|
| 小型模型调试 | 1 × RTX 3090 (24GB) | 调试 ResNet18 模型，预计 2 小时 |
| 中型模型训练 | 2 × A100 (40GB) | 训练 BERT-base，预计 12 小时 |
| 大模型微调 | 4 × A100 (80GB) | 微调 LLaMA-7B，预计 48 小时 |
| 推理服务 | 1 × V100 (32GB) | 部署在线推理服务 |

**长期占用管理**:

✅ **推荐做法**:
- 长期任务（>7天）提前向管理员申请
- 每周更新一次任务进度
- 设置合理的预计时长，系统会自动提醒

❌ **避免做法**:
- 长期占用不汇报进度
- 故意设置超长预计时长规避提醒

**示例**:
```
任务名称：LLaMA-65B 全量微调
GPU 申请：4 × A100 80GB
预计时长：14 天
进度更新：每周一提交进度报告
```

#### 13.2 任务调度最佳实践

##### 13.2.1 优先级设置指南

**优先级说明**:
- 优先级范围：0-10（数字越大优先级越高）
- 系统会优先为高优先级任务分配资源
- 相同优先级按提交时间排队

**优先级设置建议**:

| 优先级 | 适用场景 | 示例 |
|--------|---------|------|
| **9-10** | 紧急生产任务、截止日期临近 | 论文截稿前最后的实验 |
| **7-8** | 重要项目、有明确时间要求 | 项目里程碑前的训练任务 |
| **5-6** | 常规任务（默认） | 日常模型训练、数据预处理 |
| **3-4** | 低优先级任务、可等待 | 探索性实验、参数调优 |
| **0-2** | 后台任务、无时间要求 | 数据备份、模型压缩 |

✅ **推荐做法**:
- 根据任务紧急程度设置合理的优先级
- 同类型任务使用相同优先级便于管理
- 紧急任务在描述中说明原因
- 截止日期临近的任务可适当提高优先级

❌ **避免做法**:
- 所有任务都设最高优先级（会导致优先级失效）
- 故意设低优先级然后频繁催促管理员
- 不根据实际情况随意设置

**示例**:
```
任务名称：CVPR 论文实验 - 最终版
优先级：9
描述：论文截稿日期 3 月 20 日，需要完成最终实验对比

```

##### 13.2.2 资源预估方法

**GPU 显存估算**:

✅ **推荐做法**:
- 使用显存计算公式预估需求
- 预留 20% 冗余空间
- 参考类似任务的显存使用

❌ **避免做法**:
- 凭感觉估算，导致频繁调整
- 申请过多资源"以防万一"

**显存估算公式**:
```
显存需求 ≈ 模型参数 × 4 (FP32) 
         + 梯度 × 4 
         + 优化器状态 × 8 
         + 激活值 × 批次大小

简化估算（FP32 训练）:
总显存 ≈ 模型大小 × 20
```

**示例**:
```
训练 ResNet-50:
模型参数: 25M × 4 bytes = 100 MB
梯度: 25M × 4 bytes = 100 MB  
优化器状态 (Adam): 25M × 8 bytes = 200 MB
激活值 (batch=32): ~2 GB
总计: ~2.4 GB，建议申请 4GB 以上显存的 GPU
```

**运行时长预估**:

✅ **推荐做法**:
- 参考历史相似任务时长
- 先小批量测试再估算
- 预留 30% 缓冲时间
- 更新实际运行时长供下次参考

❌ **避免做法**:
- 完全不考虑时长，随便填写
- 预估过短导致任务被终止
- 预估过长占用资源太久

**时长预估表**:
```
小模型训练 (< 10M 参数):
- 单 GPU: 1-4 小时/epoch
- 数据集 10K: 约 2-8 小时完成

中模型训练 (10M-100M 参数):
- 单 GPU: 4-12 小时/epoch
- 数据集 100K: 约 1-3 天完成

大模型训练 (> 100M 参数):
- 多 GPU 并行
- 需要根据具体模型和硬件评估
```

#### 13.3 安全最佳实践

##### 13.3.1 密码管理

**强密码要求**:
- 最少 12 个字符
- 包含大写字母、小写字母、数字、特殊字符
- 不使用常见单词、生日、手机号

✅ **推荐做法**:
- 使用密码管理器（如 1Password、Bitwarden）
- 定期更换密码（建议每 90 天）
- 密码泄露后立即修改
- 使用密码短语（如 "Correct-Horse-Battery-Staple"）

❌ **避免做法**:
- 使用简单密码（如 123456、password）
- 使用个人信息（生日、手机号）
- 多个系统使用相同密码
- 将密码写在便签或文本文件中

**密码强度对比**:

| 密码示例 | 强度 | 说明 |
|---------|------|------|
| `123456` | ❌ 极弱 | 几秒内可破解 |
| `password123` | ❌ 弱 | 常见密码，易被字典攻击 |
| `ZhangSan1990` | ⚠️ 中 | 包含个人信息 |
| `Xy7!mK9#pL2@` | ✅ 强 | 随机组合，难以破解 |
| `My-Dog-Loves-2-Play!` | ✅ 强 | 密码短语，易记且安全 |

**启用 2FA 的好处**:
- 即使密码泄露，账号仍安全
- 防止未授权访问
- 符合安全合规要求

✅ **推荐做法**:
- 所有用户都应启用 2FA
- 使用专业认证器 App（Google Authenticator、Microsoft Authenticator）
- 安全保存备用码
- 手机丢失后立即使用备用码登录并重新绑定

❌ **避免做法**:
- 认为麻烦而不启用 2FA
- 将备用码保存在不安全的地方
- 手机丢失后不处理 2FA 绑定

**启用 2FA 步骤**:
```
1. 进入 设置 → 账户安全
2. 点击 "启用双因素认证"
3. 使用认证器 App 扫描二维码

5. 保存备用码（安全位置！）
6. 完成启用
```

**备用码使用场景**:
```
□ 手机丢失
□ 认证器 App 数据丢失
```

#### 13.4 性能优化最佳实践

##### 13.4.1 GPU 使用优化

**显存优化技巧**:

✅ **推荐做法**:
- 使用混合精度训练（FP16/BF16）
- 启用梯度检查点节省显存
- 合理设置批次大小
- 及时清理不再使用的变量

❌ **避免做法**:
- 盲目使用大批次大小
- 不优化直接增加 GPU 数量
- 训练脚本内存泄漏

**优化示例**:
```python
# ✅ 推荐：混合精度训练
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()
for data, target in dataloader:
    with autocast():
        output = model(data)
        loss = criterion(output, target)
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()

# ❌ 避免：不优化显存使用
for data, target in dataloader:
    output = model(data)  # 可能占用过多显存
    loss = criterion(output, target)
    loss.backward()
    optimizer.step()
```

**多 GPU 策略**:

| 场景 | 推荐策略 | 说明 |
|------|---------|------|
| 数据量大、模型小 | DataParallel | 简单易用，适合快速验证 |
| 模型大、显存不够 | DistributedDataParallel | 更高效，支持大模型 |
| 超大模型 | 模型并行 | 将模型分布到多 GPU |
| 推理服务 | 批量推理 | 提高吞吐量 |

##### 13.4.2 任务执行优化

**代码优化建议**:

✅ **推荐做法**:
- 数据预处理提前完成
- 使用高效的数据加载器
- 减少 I/O 操作
- 合理设置检查点保存频率

❌ **避免做法**:
- 每个迭代都保存检查点
- 数据加载成为瓶颈
- 不必要的打印和日志

**检查点策略**:
```python
# ✅ 推荐：合理保存检查点
if epoch % 5 == 0 or epoch == total_epochs:
    torch.save({
        'epoch': epoch,
        'model_state_dict': model.state_dict(),
        'optimizer_state_dict': optimizer.state_dict(),
    }, f'checkpoint_epoch_{epoch}.pt')

# ❌ 避免：每个迭代都保存
for batch in dataloader:
    # 训练代码...
    torch.save(model.state_dict(), 'model.pt')  # 太频繁！
```

### 第 14 章 常见问题 (FAQ)

#### 账号相关

**Q1: 忘记密码怎么办？**

A: 在登录页面点击"忘记密码"，输入注册邮箱，系统会发送重置链接。

**Q2: 如何修改用户名？**

A: 用户名注册后不可修改，如需修改请联系管理员创建新账号。

**Q3: 双因素认证失效了怎么办？**

A: 使用注册时保存的备用码登录，或联系管理员重置 2FA。

#### 服务器管理

**Q4: 添加服务器后显示离线怎么办？**

A: 
1. 检查 IP 地址和 SSH 端口是否正确
2. 检查服务器网络连接
3. 检查 SSH 服务是否运行
4. 点击"测试连接"查看详细错误信息

**Q5: 如何批量添加服务器？**

A: 使用批量导入功能，下载 CSV 模板，填写后上传。

#### GPU 管理

**Q6: 为什么申请不到 GPU？**

A: 
- 可能所有 GPU 都已分配
- 筛选条件过于严格
- 优先级较低
- 建议：放宽筛选条件、避开高峰时段、联系管理员

**Q7: GPU 分配后多久会自动释放？**

A: 默认不会自动释放，需手动释放。但系统会发送提醒：
- 分配 3 天：邮件提醒
- 分配 7 天：邮件 + 站内信提醒
- 分配 14 天：管理员介入

#### 任务管理

**Q8: 任务一直处于待处理状态怎么办？**

A: 
- 检查优先级设置
- 查看是否有可用资源
- 考虑提高优先级或联系管理员

**Q9: 任务失败后如何查看原因？**

A: 
1. 点击任务名称查看详情
2. 查看"执行日志"标签页
3. 关注错误信息 (红色高亮)
4. 根据错误信息排查问题

**Q10: 如何重新运行失败的任务？**

A: 
1. 在任务详情页点击"重新运行"
2. 确认或修改任务配置
3. 提交重新运行

#### 数据导出

**Q11: 导出的文件在哪里下载？**

A: 导出完成后，在"导出历史"中点击下载链接。

**Q12: 导出失败怎么办？**

### 第 15 章 故障排查

#### 15.1 登录问题

**无法登录**:

| 症状 | 可能原因 | 解决方案 |
|------|----------|----------|
| 提示"用户名或密码错误" | 密码错误 | 重新输入，注意大小写 |
| 提示"账号已禁用" | 账号被禁用 | 联系管理员 |

| 2FA 验证码无效 | 时间不同步 | 校准手机时间，使用备用码 |

#### 15.2 服务器连接问题

**服务器显示离线**:

| 症状 | 可能原因 | 解决方案 |
|------|----------|----------|
| 新添加服务器立即离线 | SSH 连接失败 | 检查 IP、端口、防火墙 |
| 运行中服务器突然离线 | 服务器宕机 | 检查服务器状态，重启 |
| 间歇性离线 | 网络不稳定 | 检查网络连接 |

#### 15.3 GPU 分配问题

**无法分配 GPU**:

| 症状 | 可能原因 | 解决方案 |
|------|----------|----------|
| 提示"无可用 GPU" | 全部已分配 | 等待释放或联系管理员 |
| 分配后状态不变 | 系统延迟 | 等待 30 秒刷新 |
| GPU 显示维护中 | 人工设置 | 联系管理员解除 |

#### 15.4 任务执行问题

**任务失败**:

| 错误类型 | 可能原因 | 解决方案 |
|----------|----------|----------|
| 资源不足 | GPU/CPU/内存不足 | 释放资源或等待 |
| 脚本错误 | 代码有 bug | 检查日志，修复代码 |
| 超时 | 运行时间过长 | 优化代码或增加超时 |
| 权限不足 | 文件权限问题 | 检查文件权限 |

#### 15.5 性能问题

**系统响应慢**:

| 症状 | 可能原因 | 解决方案 |
|------|----------|----------|
| 页面加载慢 | 网络问题 | 检查网络连接 |
| 列表刷新慢 | 数据量大 | 应用筛选条件 |
| 操作响应慢 | 服务器负载高 | 联系管理员 |

#### 18.4 集群预约

##### 18.4.1 功能说明

集群预约允许用户预约整个集群资源，适用于：

##### 18.4.2 新建集群预约

##### 18.4.3 AI 智能时间推荐 🆕

**功能说明**: 系统根据集群使用历史和当前状态，智能推荐最佳预约时间段。

**使用方法**:
1. 选择目标集群
2. 点击 **"获取推荐时间"** 按钮
3. 系统显示推荐时间段列表

**推荐信息**:

| 字段 | 说明 |
|------|------|
| 时间段 | 推荐的开始和结束时间 |
| 评分 | 推荐质量评分（0-100） |
| 置信度 | 推荐可靠性（高/中/低） |
| 推荐理由 | 为什么推荐这个时间段 |

**推荐示例**:
```
┌────────────────────────────────────────────────┐
│  🤖 AI 推荐时间段                               │
├────────────────────────────────────────────────┤

│  时间: 2026-03-26 02:00 - 2026-03-26 08:00    │
│  评分: 92 分                                    │
│  置信度: 高                                     │
│  理由: 凌晨时段无冲突，集群负载低               │
├────────────────────────────────────────────────┤
│  📅 备选方案 1                                  │
│  时间: 2026-03-27 22:00 - 2026-03-28 06:00    │
│  评分: 85 分                                    │
│  置信度: 中                                     │
│  理由: 夜间时段，与现有预约无冲突               │
└────────────────────────────────────────────────┘
```

**一键应用**:
- 点击推荐时间段旁的 **"应用"** 按钮
- 系统自动填充开始和结束时间

##### 18.4.4 集群预约冲突检测

系统会自动检测预约冲突：
- ✅ 无冲突 — 可直接提交

```
建议：
```

#### 18.8 最佳实践

##### 18.8.1 预约策略

| 场景 | 建议 |
|------|------|
| 大型训练任务 | 提前 2-3 天预约，选择非高峰时段（晚间/周末） |
| 紧急任务 | 联系管理员协调，或使用高优先级标记 |
| 长期项目 | 拆分为多个短期预约，便于灵活调整 |
| 不确定时长 | 预留 20% 缓冲时间，完成后及时释放 |

##### 18.8.2 避免常见问题

| 问题 | 解决方案 |
|------|----------|
| 预约冲突 | 使用 AI 时间推荐，选择无冲突时段 |
| 忘记取消 | 设置日历提醒，提前评估需求 |
| 资源不足 | 关注资源仪表盘，选择负载较低的服务器 |
| 审批延迟 | 提前预约，预留审批时间 |

##### 18.8.3 效率提升技巧

- 使用 AI 时间推荐快速找到最佳时段
- 月视图规划整体安排，日视图精确调整
- 利用批量操作处理多个预约
- 订阅日历提醒，避免遗漏重要预约

### 第 19 章 需求管理

#### 19.1 功能概述

**需求管理**页面允许所有用户提交对 LSM 系统的功能改进建议和新功能需求，帮助产品团队收集用户反馈。

**适用角色**: 所有用户

#### 19.2 查看需求列表

1. 点击左侧菜单 **"需求"** 或 **"Requirements"**
2. 页面显示已提交的需求列表，包含：
   - 需求标题
   - 提交者
   - 状态（规划中 / 开发中 / 已完成）
   - 优先级
   - 投票数

#### 19.3 提交新需求

1. 点击 **"提交需求"** 按钮
2. 填写需求表单：

| 字段 | 说明 | 示例 |
|------|------|------|
| 标题 | 简洁描述需求 | "支持 Jupyter Notebook 集成" |
| 描述 | 详细说明需求背景和期望效果 | "希望能在页面内直接启动…" |
| 优先级 | 评估对工作的影响程度 | 高 / 中 / 低 |
| 分类 | 需求所属功能模块 | GPU管理 / 任务调度 / 监控 |

3. 点击 **"提交"**

#### 19.4 为需求投票

1. 在需求列表中找到感兴趣的需求
2. 点击 **"👍 投票"** 按钮
3. 票数越高，需求被优先实现的概率越大

#### 19.5 跟踪需求进度

需求状态流转：
```
提交 → PENDING → PLANNED（规划中）→ IN_PROGRESS（开发中）→ COMPLETED（已完成）
                              ↘ REJECTED（已拒绝）
```

当需求状态变更时，提交者会收到站内通知和邮件提醒。
