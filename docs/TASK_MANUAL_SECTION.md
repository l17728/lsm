## 6.9 任务队列和调度

任务队列和调度是 LSM 系统的核心能力，确保计算任务能够高效、公平地分配到合适的服务器资源上执行。

### 6.9.2 任务队列机制

#### 队列结构

| 队列类型 | 说明 | 优先级范围 |
|----------|------|------------|
| **高优先级队列** | 紧急任务、生产任务 | CRITICAL (3) |
| **标准队列** | 常规任务、日常实验 | HIGH (2), MEDIUM (1) |
| **低优先级队列** | 后台任务、批处理任务 | LOW (0) |

#### 队列调度流程

```
任务提交 → 进入对应优先级队列 → 等待资源 → 调度器评估 → 分配资源 → 执行
              │                     │
              ▼                     ▼
         ┌─────────┐          ┌─────────────┐
         │ 高优先级 │          │ 检查资源    │
         │ 队列    │          │ 可用性      │
         ├─────────┤          ├─────────────┤
         │ 标准队列 │    →     │ 评估服务器  │
         ├─────────┤          │ 负载情况    │
         │ 低优先级 │          │ 匹配 GPU    │
         │ 队列    │          │ 需求        │
         └─────────┘          └─────────────┘
```

#### 队列管理操作

**查看队列状态**:

| 操作 | 说明 |
|------|------|
| 查看队列深度 | 显示各优先级队列的等待任务数 |
| 查看预计等待时间 | 基于当前负载估算任务开始时间 |
| 查看资源占用 | 显示当前正在执行的任务及其资源使用 |

**队列调整**（管理员）:

| 操作 | 说明 | 权限 |
|------|------|------|
| 调整任务优先级 | 改变任务在队列中的位置 | MANAGER+ |
| 暂停队列 | 暂停接受新任务 | ADMIN |
| 清空队列 | 取消所有等待中的任务 | ADMIN |

### 6.9.3 智能调度策略

LSM 系统采用 AI 驱动的智能调度，综合考虑多个因素做出最优调度决策。

#### 调度决策因素

| 因素 | 权重 | 说明 |
|------|------|------|
| **任务优先级** | 25% | 高优先级任务优先调度 |
| **资源匹配度** | 20% | 任务需求与服务器资源的匹配程度 |
| **负载均衡** | 20% | 避免单点过载，均衡分配负载 |
| **用户公平性** | 15% | 确保各用户公平获取资源 |
| **能效优化** | 10% | 优先选择能耗效率高的服务器 |
| **可靠性** | 10% | 优先选择高可靠性服务器 |

#### 调度算法详解

**1. 优先级调度 (Priority Scheduling)**

- 高优先级任务始终优先于低优先级任务
- 相同优先级按提交时间先进先出 (FIFO)
- 支持优先级抢占（需管理员配置）

```
优先级排序示例:
队列中任务: T1(P=9), T2(P=7), T3(P=6), T4(P=5), T5(P=3)

调度顺序: T1 → T2 → T3 → T4 → T5
```

**2. 资源匹配调度 (Resource Matching)**

- 根据任务 GPU 显存需求选择合适的服务器
- 考虑 CPU 核心数和内存需求
- 支持指定 GPU 型号偏好

| 任务需求 | 匹配策略 |
|----------|----------|
| GPU 显存 ≤ 16GB | 优先分配 RTX 3090、V100 |
| GPU 显存 16-40GB | 分配 A100-40GB、V100-32GB |
| GPU 显存 > 40GB | 分配 A100-80GB |
| 多卡需求 | 分配同一服务器的多块 GPU |

**3. 负载均衡调度 (Load Balancing)**

- 实时监控服务器 CPU、内存、GPU 使用率
- 避免将任务调度到过载服务器
- 支持任务迁移（仅限 PENDING 状态）

```
负载均衡示例:
服务器 A: CPU 85%, GPU 90%  → 负载过高，避免分配新任务
服务器 B: CPU 30%, GPU 20%  → 负载较低，优先分配
服务器 C: CPU 50%, GPU 60%  → 负载适中，可分配
```

**4. 公平调度 (Fair Scheduling)**

- 确保各用户公平获取资源
- 长时间等待的任务获得优先级加成
- 避免单用户独占资源

```
公平调度计算:
用户 A: 3 个任务运行中 → 降低新任务优先级
用户 B: 0 个任务运行中, 等待 2 小时 → 提升任务优先级
用户 C: 1 个任务运行中 → 正常调度
```

**5. 预测性调度 (Predictive Scheduling)**

- 基于历史数据预测负载趋势
- 提前预留资源给即将到来的高负载
- 识别即将释放的 GPU 资源

```
预测调度示例:
当前时间: 14:00
预测: 服务器 A 负载持续上升，预计 16:00 过载
决策: 将新任务调度到服务器 B，避免拥堵
```

## 6.10 任务依赖关系

任务依赖允许定义任务之间的执行顺序，确保前置任务完成后才执行后续任务。

### 6.10.1 依赖关系概述

**支持的依赖类型**:

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| **顺序依赖** | 任务 B 必须在任务 A 完成后执行 | 数据预处理 → 模型训练 |
| **并行依赖** | 任务 C 等待任务 A 和 B 都完成 | 多模型训练 → 结果合并 |
| **条件依赖** | 根据前置任务结果决定是否执行 | 训练成功 → 部署；训练失败 → 调参 |

### 6.10.2 配置任务依赖

**方式一：创建任务时指定**

```yaml
# 任务配置示例
任务名称: 模型训练
依赖任务: 
  - taskId: "data-preprocess-task-id"
    type: "sequential"  # 顺序依赖
    
任务名称: 结果评估  
依赖任务:
  - taskId: "model-train-task-id-1"
  - taskId: "model-train-task-id-2"
  依赖类型: "all"  # 所有前置任务完成
```

**方式二：在任务详情页添加依赖**

1. 进入任务详情页
2. 点击 **"添加依赖"** 按钮
3. 选择前置任务
4. 选择依赖类型
5. 点击 **"保存"**

### 6.10.3 依赖关系可视化

系统提供依赖关系图，直观展示任务间的依赖：

```
依赖关系图示例:

┌──────────────┐
│ 数据预处理    │ (已完成 ✅)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 模型训练     │ (运行中 ▶️)
└──────┬───────┘
       │
       ├─────────────────┐
       ▼                 ▼
┌──────────────┐  ┌──────────────┐
│ 模型评估     │  │ 模型部署     │
│ (等待中 ⏳)  │  │ (等待中 ⏳)  │
└──────────────┘  └──────────────┘
```

**状态说明**:

| 前置任务状态 | 后续任务状态 | 说明 |
|-------------|-------------|------|
| 待处理/运行中 | 阻塞 | 等待前置任务完成 |
| 已完成 | 就绪 | 可以开始执行 |
| 已失败 | 可配置 | 默认取消，可配置为跳过 |

### 6.10.4 依赖链管理

**最佳实践**:

- ✅ 依赖链深度建议不超过 5 层
- ✅ 避免循环依赖（A→B→C→A）
- ✅ 为关键依赖设置超时和告警
- ✅ 定期检查并清理无效依赖

**处理依赖冲突**:

```
检测到循环依赖时的处理:
1. 系统自动检测并阻止创建循环依赖
2. 提示用户修改依赖关系
3. 建议将循环拆分为独立的任务流
```

### 6.10.5 依赖任务执行策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **严格依赖** | 前置任务必须成功 | 关键任务链 |
| **宽松依赖** | 前置任务完成即可（无论成功失败） | 可容错的任务 |
| **条件依赖** | 根据前置任务输出决定执行 | 分支流程 |

## 6.11 任务优先级设置详解

### 6.11.1 优先级体系

LSM 系统采用四级优先级体系：

| 优先级 | 数值 | 图标 | 说明 | 调度权重 |
|--------|------|------|------|----------|
| **CRITICAL** | 3 | 🔴 | 紧急任务 | 最高，优先调度 |
| **HIGH** | 2 | 🟠 | 高优先级任务 | 高 |
| **MEDIUM** | 1 | 🟡 | 普通任务（默认） | 中等 |
| **LOW** | 0 | 🟢 | 低优先级任务 | 低 |

### 6.11.2 优先级设置规则

**自动优先级调整**:

系统会根据以下因素自动调整任务优先级：

| 因素 | 调整规则 |
|------|----------|
| 等待时间 | 每等待 1 小时，优先级 +0.1 |
| 用户历史成功率 | 成功率 < 50%，优先级 +0.2 |
| 管理员标记 | 管理员可手动调整优先级 |

**优先级限制**:

| 用户角色 | 可设置优先级范围 | 说明 |
|----------|------------------|------|
| USER | LOW, MEDIUM | 普通用户 |
| MANAGER | LOW, MEDIUM, HIGH | 管理员 |
| ADMIN | 所有优先级 | 系统管理员 |

### 6.11.3 优先级使用建议

**场景映射**:

```yaml
# 紧急任务示例
任务: 论文截稿前最终实验
优先级: CRITICAL
说明: 论文截稿日期临近，必须立即执行

# 高优先级任务示例
任务: 生产环境模型更新
优先级: HIGH
说明: 需要尽快完成，但有缓冲时间

# 普通任务示例
任务: 日常模型训练
优先级: MEDIUM
说明: 按正常流程排队执行

# 低优先级任务示例
任务: 历史数据归档
优先级: LOW
说明: 后台任务，资源空闲时执行
```

**优先级设置检查清单**:

```
□ 任务是否有明确的截止时间？
□ 任务是否影响他人工作？
□ 是否需要占用大量资源？
□ 是否与重要项目相关？

如果有 2 个以上"是"，考虑提高优先级
```

### 6.12.2 状态转换条件

| 当前状态 | 目标状态 | 触发条件 | 权限要求 |
|----------|----------|----------|----------|
| PENDING | RUNNING | 调度器分配资源并启动 | 系统 |

| RUNNING | COMPLETED | 任务执行成功完成 | 系统 |
| RUNNING | FAILED | 任务执行过程中出错 | 系统 |
| RUNNING | CANCELLED | 用户主动取消 | 任务所有者 |
| COMPLETED | PENDING | 重新运行任务 | 任务所有者 |
| FAILED | PENDING | 重新运行任务 | 任务所有者 |
| CANCELLED | PENDING | 重新运行任务 | 任务所有者 |

### 6.12.3 状态相关操作

**PENDING 状态可执行操作**:

| 操作 | 说明 | 权限 |
|------|------|------|
| 编辑任务 | 修改任务配置 | 任务所有者 |
| 调整优先级 | 改变调度顺序 | 任务所有者/MANAGER+ |
| 取消任务 | 取消任务执行 | 任务所有者 |

| 操作 | 说明 | 权限 |
|------|------|------|
| 查看日志 | 实时查看执行日志 | 任务所有者 |
| 查看监控 | 查看资源使用情况 | 任务所有者 |
| 取消任务 | 终止任务执行 | 任务所有者 |

**COMPLETED/FAILED/CANCELLED 状态可执行操作**:

| 操作 | 说明 | 权限 |
|------|------|------|
| 查看结果 | 查看任务输出 | 任务所有者 |
| 下载日志 | 下载执行日志文件 | 任务所有者 |
| 重新运行 | 创建相同配置的新任务 | 任务所有者 |
| 删除任务 | 删除任务记录 | 任务所有者/ADMIN |

### 6.12.4 状态通知配置

用户可为不同状态变更设置通知：

| 状态变更 | 默认通知 | 可配置 |
|----------|----------|--------|

### 6.13.1 任务创建最佳实践

```

✅ NLP-BERT-微调-医学语料-v2
```

### 6.13.2 资源申请技巧

**GPU 数量选择指南**:

| 模型规模 | 推荐配置 | 说明 |
|----------|----------|------|
| 小模型 (< 10M 参数) | 1 GPU | 单卡即可满足 |
| 中型模型 (10M-100M) | 1-2 GPU | 可考虑数据并行 |
| 大型模型 (100M-1B) | 2-4 GPU | 数据并行 + 模型并行 |
| 超大模型 (> 1B) | 4-8 GPU | 需要分布式训练 |

**显存估算公式**:

```
显存需求 ≈ 模型参数 × 4 (FP32) + 梯度 × 4 + 优化器状态 × 8 + 激活值 × batch_size

简化估算:
训练 (FP32): 模型大小 × 20
训练 (FP16): 模型大小 × 12
推理: 模型大小 × 2
```

### 6.13.4 任务监控技巧

**实时监控要点**:

| 监控项 | 关注阈值 | 处理建议 |
|--------|----------|----------|
| GPU 使用率 | < 30% 或 > 95% | 检查是否有死循环或优化代码 |
| GPU 显存 | > 90% | 减小 batch_size |
| GPU 温度 | > 85°C | 检查散热或降低负载 |
| 训练损失 | 不下降 | 检查学习率或数据 |
| 进度停滞 | > 30 分钟无更新 | 检查日志是否有问题 |

### 6.13.6 常见问题与解决

**问题 1: 任务长时间处于 PENDING 状态**

```
诊断步骤:
1. 检查任务优先级是否过低
2. 查看当前资源使用情况
3. 检查任务资源需求是否过高

解决方案:
- 提高任务优先级
- 减少资源需求
- 等待资源释放
- 联系管理员协调
```

**问题 2: 任务频繁失败**

```
诊断步骤:
1. 查看错误日志
2. 检查资源是否充足
3. 验证脚本正确性

常见失败原因:
- GPU 显存不足 (OOM) → 减小 batch_size
- 数据路径错误 → 检查路径配置
- 权限问题 → 检查文件权限
- 超时 → 申请更长执行时间
```

**问题 3: 调度效率低下**

```
优化建议:
1. 使用任务模板减少重复配置
2. 合理设置优先级
3. 避免资源浪费（申请过多但实际使用少）
4. 利用定时任务错峰运行
5. 定期清理已完成任务
```
