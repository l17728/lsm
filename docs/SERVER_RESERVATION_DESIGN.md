## 1. 功能概述

### 1.1 目标用户

| 用户类型 | 使用场景 | 核心需求 |
|---------|---------|---------|
| 研究生/博士生 | 日常实验、模型训练 | 按需申请 GPU，短期使用 |
| 研究员/教授 | 大型实验、长期项目 | 整服务器预约，稳定资源 |
| 实验室管理员 | 资源管理、调度优化 | 审批、配额管理、利用率监控 |

### 1.2 核心价值

- **提高资源利用率**: 通过预约机制减少资源闲置
- **公平分配**: 透明的预约规则，避免资源抢占
- **简化管理**: 可视化日历，一键预约
- **减少冲突**: 自动冲突检测，智能建议

## 2. 申请模式设计

### 2.1 四种申请模式

| 模式 | 说明 | 适用场景 | 优点 | 缺点 |
|------|------|---------|------|------|
| **单 GPU** | 申请指定服务器的单个 GPU | 小型实验、调试 | 灵活、等待时间短 | 需要手动管理多个 GPU |
| **多 GPU** | 申请同一服务器的多个 GPU | 分布式训练、大模型 | 同机通信效率高 | 需服务器有足够空闲 GPU |
| **整服务器** | 申请服务器全部资源 | 多卡训练、独占需求 | 完全隔离、无干扰 | 等待时间长、可能浪费 |
| **时间段预约** | 提前预约未来时间段 | 定期实验、教学使用 | 计划性强、可提前安排 | 需要提前规划 |

### 2.2 申请流程

```
用户发起申请
    │
    ├── 选择申请模式
    │       ├── 单 GPU → 选择服务器 → 选择 GPU
    │       ├── 多 GPU → 选择服务器 → 选择 GPU 数量
    │       ├── 整服务器 → 选择服务器
    │       └── 时间段预约 → 选择时间段 → 选择资源
    │
    ├── 系统检测冲突
    │       ├── 无冲突 → 进入审批（如需）
    │       └── 有冲突 → 显示冲突信息 → 建议替代方案
    │
    ├── 审批流程（可配置）
    │       ├── 无需审批 → 直接生效
    │       ├── 单级审批 → 管理员审批
    │       └── 多级审批 → 逐级审批
    │
    └── 预约生效
            ├── 发送通知
            ├── 添加日历
            └── 到期自动释放
```

## 3. 核心功能模块

### 3.1 预约日历

**功能描述**: 可视化展示资源占用情况

**视图模式**:
- 日视图: 展示当天各服务器/GPU 的占用情况
- 周视图: 展示一周内的资源预约情况
- 月视图: 展示一个月内的资源使用趋势
- 资源视图: 按服务器/GPU 维度展示

**交互功能**:
- 点击空白区域快速创建预约
- 拖拽调整预约时间
- 悬停查看预约详情
- 筛选特定服务器/用户

### 3.2 冲突检测

**检测规则**:
1. 时间重叠检测: 新预约时间段与已有预约重叠
2. 资源独占检测: 整服务器预约与其他 GPU 预约冲突
3. 配额限制检测: 用户/角色配额超限
4. 维护窗口检测: 服务器处于维护状态

**冲突处理**:
- 显示冲突详情
- 建议最近的可用时间段
- 建议替代服务器

### 3.3 自动释放

**释放触发**:
- 预约时间结束
- 任务执行完成
- 用户主动释放
- 超时未使用（可配置）

**释放流程**:
1. 发送到期提醒（提前 15 分钟）
2. 时间到达，自动释放
3. 更新资源状态
4. 通知用户释放完成

### 3.4 通知提醒

**通知场景**:
| 场景 | 通知对象 | 通知渠道 |
|------|---------|---------|
| 预约创建成功 | 申请人 | 站内信/邮件 |
| 预约即将开始 | 申请人 | 站内信/邮件/钉钉 |
| 预约即将结束 | 申请人 | 站内信 |
| 审批请求 | 审批人 | 站内信/邮件/钉钉 |
| 审批结果 | 申请人 | 站内信/邮件 |
| 冲突警告 | 申请人 | 站内信 |

## 4. 权限与配额

### 4.1 权限矩阵

| 操作 | USER | MANAGER | ADMIN |
|------|------|---------|-------|
| 创建预约 | ✅ | ✅ | ✅ |
| 查看自己的预约 | ✅ | ✅ | ✅ |
| 查看所有预约 | ❌ | ✅ | ✅ |
| 取消自己的预约 | ✅ | ✅ | ✅ |
| 取消他人预约 | ❌ | ✅ | ✅ |
| 审批预约 | ❌ | ✅ | ✅ |
| 配置配额 | ❌ | ❌ | ✅ |
| 管理服务器 | ❌ | ✅ | ✅ |

### 4.2 配额设计

**配额类型**:
- 单次预约时长限制: 如最多预约 72 小时
- 单次 GPU 数量限制: 如最多申请 4 个 GPU
- 同时有效预约数: 如最多 3 个待执行预约
- 每周/每月总时长限制: 如每周最多 100 小时

**配额层级**:
1. 用户级配额: 针对单个用户
2. 角色级配额: 针对用户角色（学生/研究员）
3. 全局配额: 系统默认限制

## 5. 调度算法

### 5.1 资源分配策略

**优先级排序**:
1. 紧急任务（系统标记）
2. 教学任务（预约时间段）
3. 科研任务（按申请时间）
4. 个人实验（按申请时间）

**分配算法**:
```
function allocateResources(request):
    // 1. 检查配额
    if !checkQuota(request.user, request.resources):
        return QUOTA_EXCEEDED
    
    // 2. 检查冲突
    conflicts = findConflicts(request)
    if conflicts:
        return suggestAlternatives(conflicts)
    
    // 3. 选择最优资源
    if request.mode == WHOLE_SERVER:
        server = findBestAvailableServer(request.timeRange)
    elif request.mode == MULTI_GPU:
        server = findServerWithEnoughGPUs(request.gpuCount, request.timeRange)
    else:
        gpu = findBestAvailableGPU(request.serverId, request.timeRange)
    
    // 4. 创建预约
    return createReservation(request, resources)
```

### 5.2 碎片优化

**问题**: 单 GPU 预约导致服务器 GPU 碎片化

**解决方案**:
1. **智能建议**: 当用户申请单 GPU 时，建议选择已有预约的服务器剩余 GPU
2. **合并策略**: 自动建议将多个单 GPU 预约合并到同一服务器
3. **预留策略**: 为整服务器预约预留部分服务器

## 8. 风险与限制

### 8.1 已知风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 并发预约冲突 | 可能导致重复预约 | 乐观锁 + 事务 |
| 资源滥用 | 用户长时间占用 | 配额限制 + 自动释放 |
| 用户不释放 | 资源闲置 | 超时自动释放 + 信用机制 |

### 8.2 功能限制

- 不支持跨服务器 GPU 预约
- 时间段预约最短 1 小时
- 整服务器预约需提前 24 小时申请
