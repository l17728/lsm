## 1️⃣ 需求理解流程

### 步骤

```
1. 接收用户需求
   ↓
2. 分析需求类型
   ↓
3. 确认需求细节（如需要）
   ↓
4. 复述需求确认理解
   ↓
5. 提出执行方案
   ↓
6. 用户确认后执行
```

### 需求类型分类

| 类型 | 示例 | 处理方式 |
|------|------|----------|
| 环境搭建 | "搭建文档门户" | 技术方案 → 实施 → 测试 → 文档 |
| 功能开发 | "实现任务执行引擎" | 设计 → 编码 → 测试 → 提交 |

| 文档编写 | "写会议总结" | 收集信息 → 编写 → 审核 → 归档 |
| 问题排查 | "服务启动失败" | 诊断 → 修复 → 验证 → 记录 |

### 实例分析

**用户**: "请把环境搭建起来可以通过 web 访问，另外请将所有文档都放在 web 工具的页面可以点击下载或者浏览。"

1. **需求类型**: 环境搭建
2. **核心需求**:
   - Web 可访问
   - 文档浏览功能
   - 文档下载功能
3. **技术方案**: Node.js + Express + Markdown 渲染
4. **执行步骤**:
   - 创建项目
   - 安装依赖
   - 编写代码

## 5️⃣ Git 工作流程

### 分支策略

```
main (生产)
  ↑
develop (开发)
  ↑
feature/* (功能)
```

### 提交流程

```bash
# 1. 创建功能分支
git checkout develop
git checkout -b feature/xxx

# 2. 开发并提交
# ... 编写代码 ...
git add .
git commit -m "feat: 实现 xxx 功能"

# 3. 推送分支
git push origin feature/xxx

# 4. 创建 PR（虚拟）
# AI 直接合并到 develop
git checkout develop
git merge feature/xxx

# 5. 删除功能分支
git branch -d feature/xxx
```

### 提交信息规范

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型**:
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `test`: 测试相关
- `refactor`: 重构
- `chore`: 构建/配置
